Getting Started

Preparation

  1. Make sure your system has installed Node.js.
  2. Open a terminal window and go to the Sources directory.
  3. Run the npm install command. This command will load all necessary dependencies specified in the package.json file.
  4. Then run the npm install -g @angular/cli command to install Angular CLI (if not yet installed).

Build Project

To start a local HTTP server and runs the build process with change tracking run the command:

ng serve

To build for production, run the command:

ng build --prod

After executing the command, the compiled project will be located in the dist directory.

For a complete list of available Angular CLI commands, go to this link.

Customization

How to change logo?

  • Desktop header - In the projects/storefront/src/app/modules/header/components/logo/logo.component.html file, find comments <!-- logo --> and <!-- logo / end --> and replace code between them with your logo image.
  • Mobile header - In the projects/storefront/src/app/modules/mobile/components/mobile-logo/mobile-logo.component.html file, find comments <!-- mobile-logo --> and <!-- mobile-logo / end --> and replace code between them with your logo image.

How to change font?

  1. Open the projects/storefront/src/index.html file.
  2. Find and replace with the desired font:
    <!-- fonts -->
<link rel="stylesheet" href="https://fonts.googleapis.com/css?family=Roboto:400,400i,500,500i,700,700i">
  3. Open the projects/storefront/src/scss/_custom-variables.scss file.
  4. And define a special variable, as in the example below: 
    $body-font-family: 'YOUR_FONT_NAME';

How to change theme color scheme?

Predefined color schemes

  • To compile a template with one of the predefined color schemes, open the projects/storefront/src/scss/_custom-variables.scss file.
  • Then find and uncomment one of the following lines: 
    //@import 'themes/theme-red/theme-variables';
//@import 'themes/theme-blue/theme-variables';
//@import 'themes/theme-green/theme-variables';
//@import 'themes/theme-orange/theme-variables';
//@import 'themes/theme-violet/theme-variables';
  • After that, you need to rebuild the template.

Your own color scheme

  1. Open the projects/storefront/src/scss/_custom-variables.scss file.
  2. Find the code below:
    // Change main theme color
//$theme-scheme: (
//    main:  #584d8c, // main theme color
//    muted: #cdc8e5, // muted text color
//);
  3. Uncomment these lines and replace the colors with the ones you need.

How to remove/change copyright notice?

  1. Open the projects/storefront/src/app/modules/footer/footer.component.html file.
  2. Find comments <!-- copyright --> and <!-- copyright / end -->, then replace code between them with your copyright.

How to change layout and header variant?

  1. Open the projects/storefront/src/app/app-routing.module.ts file.
  2. Find the following route:
    const routes: Routes = [
    // ...
    {
        path: '',
        component: RootComponent,
        data: {
            /**
             * Desktop header layout, one of:
             * - spaceship/{one, two, three}
             * - classic/{one, two, three, four, five}
             */
            desktopHeader: 'spaceship/one',
            /**
             * Mobile header layout, one of:
             * - one
             * - two
             */
            mobileHeader: 'one',
        },
        children: [
            // ...
        ],
    },
];
  3. Then change the value of the data.desktopHeader and data.mobileHeader properties.

How to change PayPal API key?

  1. Open the projects/storefront/src/environments/environment.ts file if you want to change the API key for the development environment or projects/storefront/src/environments/environment.prod.ts for production environment.
  2. Then set your API key in the paypalClientId property of the environment object:
    export const environment = {
    // ...
    paypalClientId: 'YOUR_PAYPAL_CLIENT_ID',
    // ...
};

How does LTR/RTL support work?

The direction is controlled using two scss variables:

  • $direction - indicates direction. Possible values: ltr or rtl.
  • $both-directions - if true compiles styles for two directions, the $direction variable will be ignored. Possible values: true or false.

Some "magic" variables are also used, which change their value depending on the given direction value. The names of these variables are the same as the names of the Logical Properties (for example, variable $margin-inline-start). These variables should only be used in the direction mixin.

This example demonstrates how it works:

.alert {
    @include direction {
        #{$margin-inline-start}: 20px;
    }
}

Will be compiled to:

  • If $direction == ltr and $both-directions == false: 

    .alert {
    margin-left: 20px;
}

  • If $direction == rtl and $both-directions == false:

    .alert {
    margin-right: 20px;
}

  • If $both-directions == true:

    [dir=ltr] .alert {
    margin-left: 20px;
}
[dir=rtl] .alert {
    margin-right: 20px;
}

What if you want to use only one direction?

Open projects/storefront/src/scss/_custom-variables.scss, and define next variables:

$direction:       ltr;   // specify the direction you need (ltr or rtl)
$both-directions: false; // set to false to disable style compilation
                         // for two directions

Translation

The language configuration is in the projects/storefront/src/app/app.module.ts module.

Find the following piece of code and set the default language and an array of available language: 

LanguageModule.config({
    default: 'en',
    languages: [
        {
            code: 'en',
            name: 'English',
            image: 'assets/images/languages/language-1.png',
            direction: 'ltr',
        },
        {
            code: 'ru',
            name: 'Russian',
            image: 'assets/images/languages/language-2.png',
            direction: 'ltr',
        },
    ],
}),
  • default — The language code that will be used by default.
  • languages — Array of available languages.

How to translate template?

  1. Go to projects/storefront/src/assets/i18n/ directory.
  2. Then copy the en.json file and specify its name, for example [LANGUAGE_CODE].json.
  3. Open the copied file and translate all entries.
  4. Add a new language to the configuration.
    projects/storefront/src/app/app.module.ts 
    LanguageModule.config({
    default: 'en',
    languages: [
        {
            code: 'en',
            name: 'English',
            image: 'assets/images/languages/language-1.png',
            direction: 'ltr',
        },
        {
            code: LANGUAGE_CODE,
            name: LANGUAGE_NAME,
            image: LANGUAGE_ICON, // path to the language icon
            direction: LANGUAGE_DIRECTION, // ltr or rtl
        },
    ],
}),

Integration

First of all, you should understand that the process of integrating a template with your server is likely to be quite time-consuming and complicated. This is due to the fact that in the world of online stores there are no uniform standards. Some use integers as identifiers, others use UUID strings, and the third use something else. Nevertheless, I took some steps to simplify this process as much as possible. This will be discussed in this section.

API module

By default, the template uses "Fake Server" as the data source. You can find out more about the fake server below. To replace the “Fake Server” calls with real API calls, you need to create a module that will contain services that interact with the server.

  1. Generate a new API module by running the following command:
    ng generate module --project=storefront api/rest-api
  2. Then you need to implement all abstract services listed in the projects/storefront/src/app/api/base directory. You can use FakeApiModule and its services as an example (projects/storefront/src/app/api/fake-api).
  3. When done, got to the AppModule (projects/storefront/src/app/app.module.ts) and replace the FakeApiModule with yours (RestApiModule).

URL service

The URL service is designed to generate the URLs used in the template (projects/storefront/src/app/services/url.service.ts). You can use it to quickly change the URLs of products, categories, etc. For example, this is how you can get the product URL containing its identifier: 

export class UrlService {
    // ...
    product(product: Product): string {
        return `/shop/products/${product.id}`;
    }
    // ...
}

Remember to make the appropriate changes to the routes and components.

What if my product fields are different from those used in the template?

Well, I can offer several solutions to this problem (they are sorted by complexity, from simple to complex):

  • Property mapping;
  • Custom fields;
  • Modify product interface.

Property mapping

This method involves converting data when it is received from the server:

@Injectable()
export class RestShopApi extends ShopApi {
    // ...
    getProductBySlug(slug: string): Observable<Product> {
        const url = `https://example.com/api/products/${slug}.json`;

        return this.http.get<Product>(url).pipe(
            map(product => ({
                id: product.productId,
                name: product.description,
                images: product.pictures,
                ...
            })),
        );
    }
    // ...
}

Custom fields

If there is no suitable property in the product interface, you can use custom fields:

@Injectable()
export class RestShopApi extends ShopApi {
    // ...
    getProductBySlug(slug: string): Observable<Product> {
        const url = `https://example.com/api/products/${slug}.json`;

        return this.http.get<Product>(url).pipe(
            map(product => ({
                ...
                customFields: {
                    isbn: product.ISBN,
                },
            })),
        );
    }
    // ...
}

Modify product interface

And finally, the last and most labor-intensive way. Change the product interface and everything related to it throughout the template.

What is a fake server and why is it needed?

I think this will surprise many, but, as practice shows, this issue also needs to be clarified.

Typically, an Angular application generates HTML pages based on dynamic data received from the server. While the Angular template comes without server-side code. However, to demonstrate the capabilities of the template, we still need something that will process requests and serve as a data provider for the template, and it is for these purposes that I created a fake server that imitates the operation of a real server. You should not use it (and everything related to it) in a real application, it is only needed to demonstrate the capabilities of the template.

All code associated with a fake server is located in the projects/storefront/src/fake-server directory. You can remove it.
